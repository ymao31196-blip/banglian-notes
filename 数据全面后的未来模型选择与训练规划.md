# 数据全面后的未来模型选择与训练规划

## 一、总体定位

后续工作的重心应从“短期功率预测验证”转向“气象预报驱动的未来几日功率预测”。短期预测实验已经证明了功率口径、时间对齐、特征构造、模型训练和评价流程可以跑通，但它不是最终业务主线。未来几日预测的核心输入必须是未来气象预报数据，尤其是未来风速、风向及其变化。

从企业自有模型建设角度看，合理路线不是完全从零发明模型，也不是直接套用开源模型，而是：

```text
成熟模型方法 + 企业自有数据 + 场站业务规则 + 多源气象融合 + 持续校准闭环
```

也就是说，LightGBM、XGBoost、TFT、PatchTST、NHITS、NBEATSx、AutoGluon、WindFM 等可以作为技术底座，但真正形成企业专用能力的是数据口径、特征体系、约束规则、融合策略、评价标准和持续更新机制。

通俗理解：模型本身只是发动机，企业自己的数据和业务规则才是燃料、道路和驾驶策略。只换一个高级模型，不处理数据和业务口径，通常不会得到稳定提升。

## 二、目标任务定义

### 1. 核心预测任务

未来模型建议围绕以下任务定义：

```text
输入：某个起报时间 init_time 下的未来气象预报、历史功率、历史实测气象、场站基础信息
输出：未来多个有效时刻 valid_time 的场站功率预测
```

建议优先覆盖：

- 未来 15 分钟至 4 小时：超短期预测。
- 未来 1 天：日前预测。
- 未来 2 至 3 天：多日滚动预测。
- 未来 7 天：中期趋势参考，可作为扩展目标。

其中，当前项目更应优先做：

```text
未来 1 天至 3 天，15 分钟粒度或 1 小时粒度功率预测
```

原因是企业气象预报数据本身就是面向未来时刻的预报文件，如果只做 15 分钟短期预测，反而没有充分利用气象预报数据的价值。

### 2. 预测对象

预测对象应为场站级功率：

```text
power_mw = clip(-(311 + 312 + 313 + 314), 0, 76)
```

其中 `76MW` 为当前使用的苗庄装机容量。后续如果企业确认实际考核口径、装机容量或功率点位不同，应以企业确认口径为准。

### 3. 时间粒度

建议分两个层级：

- 训练和精细预测：15 分钟粒度。
- 汇报和业务展示：可同时提供 15 分钟、1 小时聚合结果。

如果未来功率数据只有 15 分钟粒度，则不能训练真正的 5 分钟预测模型。5 分钟预测需要 1 分钟或 5 分钟功率标签。

## 三、数据全面后的数据体系

### 1. 必要数据

正式建模至少需要以下数据：

| 数据类型 | 作用 | 说明 |
| --- | --- | --- |
| 场站实发功率 | 训练标签 | 最关键的目标值，决定模型学什么 |
| 多源气象预报 | 未来预测核心输入 | EC、华为、盘古等不同来源 |
| 历史实测风速风向 | 当前状态和历史状态输入 | 只能使用预测时刻之前的数据 |
| 场站基础信息 | 物理约束和特征补充 | 坐标、装机、风机数量、轮毂高度、机型 |
| 风机状态/停机/检修 | 样本筛选和容量修正 | 避免把非正常发电当成气象规律 |
| 限电/弃风信息 | 标签解释和业务修正 | 大风低功率可能来自限电而不是模型错误 |
| 功率质量标记 | 异常识别 | 用于坏点过滤或样本权重 |
| 预测发布记录 | 持续校准 | 保存每次预测值，等待实测回来后评估 |

### 2. 数据时间字段

气象预报建模必须区分三个时间概念：

| 字段 | 含义 | 举例 |
| --- | --- | --- |
| `init_time` | 气象预报起报时间 | 2025-01-01 12:00 |
| `valid_time` | 预报对应的未来有效时间 | 2025-01-02 08:15 |
| `lead_time` | 预报提前量 | 20.25 小时 |

这是未来几日预测中最容易出错的地方。模型训练时，特征必须来自 `init_time` 时已经可获得的预报，标签是 `valid_time` 对应的实测功率。

通俗理解：如果 1 月 1 日 12 点发布预测，预测 1 月 2 日 8 点功率，那么模型只能用 1 月 1 日 12 点这次起报里对 1 月 2 日 8 点的天气预报，不能用 1 月 2 日 8 点之后才知道的真实天气。

### 3. 数据对齐原则

后续数据处理应坚持以下规则：

- 所有时间戳统一时区和格式。
- 气象预报按 `init_time + valid_time + lead_time` 建表。
- 功率标签按 `valid_time` 对齐。
- 历史功率和历史实测气象只能取 `init_time` 之前的数据。
- 多源气象预报必须对齐到同一 `valid_time`。
- 缺失气象源不能简单删除全部样本，应保留缺失标记。
- 限电、停机、检修样本要单独标记，必要时分开训练或剔除。

## 四、多源气象预报数据使用方法

多源气象预报是未来模型提升的关键。所谓多源，可以包括 EC、华为、盘古或其他数值天气预报/AI 天气预报来源。

多源气象不能简单理解为“文件更多”，而应理解为“多个预报系统对未来天气的不同判断”。不同气象源在不同季节、不同提前量、不同风速区间、不同风向区间可能表现不同。

### 1. 方法一：多源直接作为特征输入

最直接的做法是把不同来源的同类气象变量都作为特征输入模型。

例如同一个 `valid_time`：

```text
EC_ws100
HW_ws100
PG_ws100
EC_wd100_sin
EC_wd100_cos
HW_wd100_sin
HW_wd100_cos
PG_wd100_sin
PG_wd100_cos
```

模型会自动学习不同气象源和功率之间的关系。

优点：

- 实现简单。
- 适合 LightGBM、XGBoost 等表格模型。
- 可以快速验证多源数据是否有收益。

缺点：

- 模型解释相对粗。
- 当某个气象源缺失时，需要额外处理。
- 不同来源变量命名、空间分辨率、时间分辨率必须先统一。

### 2. 方法二：构造多源统计特征

除了保留每个气象源原始变量，还应构造多源统计特征。

以 100 米风速为例：

```text
ws100_mean = mean(EC_ws100, HW_ws100, PG_ws100)
ws100_std  = std(EC_ws100, HW_ws100, PG_ws100)
ws100_min  = min(EC_ws100, HW_ws100, PG_ws100)
ws100_max  = max(EC_ws100, HW_ws100, PG_ws100)
ws100_range = max - min
```

这些特征的意义是：

- 多源均值表示综合预报。
- 多源标准差表示预报不确定性。
- 多源最大最小值表示不同气象源分歧范围。

通俗理解：如果 EC、华为、盘古都认为未来风速差不多，预测可信度更高；如果三者差异很大，说明天气过程不确定，功率预测也更容易错。

### 3. 方法三：构造多源差异特征

可以显式构造不同气象源之间的差异：

```text
EC_ws100 - HW_ws100
PG_ws100 - EC_ws100
PG_ws100 - HW_ws100
EC_wd100_angle_diff_HW
PG_wd100_angle_diff_EC
```

风向差异不能直接用角度相减，应考虑 0 度和 360 度是相邻方向，建议用 sin/cos 或圆形距离计算。

差异特征可以帮助模型识别“预报不一致时的风险”。例如某些风向下 EC 总是偏强，某些提前量下盘古更稳定，这些规律都可以通过差异特征被模型学习。

### 4. 方法四：单源模型分别训练，再做融合

更稳健的企业级做法是每个气象源先单独训练一个模型：

```text
EC 气象预报 -> EC 功率预测模型
HW 气象预报 -> HW 功率预测模型
PG 气象预报 -> PG 功率预测模型
```

然后再训练一个融合模型：

```text
最终预测 = f(EC预测, HW预测, PG预测, lead_time, season, wind_speed, wind_direction)
```

融合方式可以从简单到复杂：

- 固定加权平均。
- 按提前量分段加权。
- 按月份、风速区间、风向区间动态加权。
- 用 LightGBM/XGBoost 训练二级融合模型。
- 用最近一段时间误差动态更新权重。

通俗理解：不是提前假设哪个气象源最好，而是让历史数据告诉我们“什么情况下哪个源更靠谱”。

### 5. 方法五：多源气象可靠性评分

持续运行后，可以给每个气象源建立可靠性评分：

```text
source_score = 最近一段时间该气象源在该场站、该提前量、该风速区间的误差表现
```

例如：

- EC 在 0-24 小时提前量表现最好。
- 盘古在 24-72 小时表现更稳。
- 华为在某些强风过程表现更好。

这些评分可以作为融合模型的输入，也可以直接用于动态权重。

### 6. 多源气象使用的推荐路线

建议按阶段推进：

| 阶段 | 方法 | 目的 |
| --- | --- | --- |
| 第一阶段 | 多源变量直接拼接 + 多源统计特征 | 快速验证多源收益 |
| 第二阶段 | 单源模型分别训练 | 分析每个气象源贡献 |
| 第三阶段 | 二级融合模型 | 提升整体稳定性 |
| 第四阶段 | 动态权重和可靠性评分 | 形成持续校准能力 |

## 五、业务特征定制

企业专用模型的重要差异不只是模型结构，而是业务特征。以下特征建议作为正式模型的重点。

### 1. 气象基础特征

每个气象源可构造：

- U/V 风分量。
- 风速 `sqrt(u^2 + v^2)`。
- 风向 sin/cos。
- 10 米风速、100 米风速、轮毂高度附近风速。
- 阵风。
- 温度。
- 露点温度。
- 气压。
- 辐照。
- 降水。
- 湿度或由温度、露点推导的湿度。

### 2. 风电专用气象特征

建议重点构造：

```text
风速平方 ws^2
风速三次方 ws^3
不同高度风切变 shear
风速变化量 delta_ws
风向变化量 delta_wd
未来窗口平均风速
未来窗口最大风速
未来窗口风速爬坡
多源风速离散度
多源风向离散度
```

风电功率与风速近似存在三次方关系，因此 `ws^2`、`ws^3` 对树模型非常重要。

### 3. 历史功率特征

历史功率仍然有价值，尤其是日内滚动预测中：

- 当前功率。
- 15min、30min、1h、2h、4h、24h 滞后功率。
- 过去 1h、4h、24h 功率均值。
- 过去 1h、4h、24h 功率标准差。
- 功率爬坡率。
- 最近是否发生快速升降。

但需要注意：未来几日预测时，随着 `lead_time` 增加，当前功率的作用会下降，未来气象预报的作用会上升。

### 4. 时间周期特征

建议加入：

- 小时 sin/cos。
- 年内日 sin/cos。
- 月份。
- 季节。
- 工作日/周末。
- 节假日，如果业务上有调度影响。

风电不像负荷预测那样强依赖工作日，但季节和日内稳定度仍可能影响模型。

### 5. 场站基础特征

如果后续扩展到多个场站，建议加入：

- 场站经纬度。
- 装机容量。
- 风机数量。
- 机型。
- 轮毂高度。
- 叶轮直径。
- 场站海拔。
- 地形粗糙度或地形类别。
- 场站所属区域。

单场站时这些特征变化不大，但多场站建模时非常重要。

### 6. 运行状态特征

建议企业提供或后续接入：

- 可用风机数量。
- 停机风机数量。
- 检修标记。
- 限电标记。
- 弃风标记。
- 并网状态。
- 通信异常标记。
- 功率质量标记。

如果没有这些特征，模型会把“停机、限电导致的低功率”误认为“正常气象条件下的低发电能力”，从而污染训练。

## 六、物理约束方法

物理约束应分阶段加入，不建议一开始堆太复杂。合理顺序是：先保证基本约束，再加入业务约束，最后加入更精细的物理规律。

### 1. 第一层：输出上下限约束

这一层应从一开始就加入：

```text
0 <= power_pred <= installed_capacity
```

对苗庄当前口径：

```text
0 <= power_pred <= 76MW
```

这是最基本的物理约束，适用于所有模型。

### 2. 第二层：动态容量约束

如果知道停机、检修、限电或可用风机数量，应使用动态可用容量：

```text
available_capacity = installed_capacity * available_turbine_ratio
0 <= power_pred <= available_capacity
```

例如 19 台风机中有 3 台检修，则最大可发功率不应仍按全容量处理。

如果存在限电指令，则预测目标应区分：

- 自然可发功率。
- 实际并网功率。

这是风电预测中非常关键的业务口径问题。

### 3. 第三层：风机功率曲线约束

风机通常存在：

- 切入风速：低于该风速基本不发电。
- 额定风速：达到后接近额定功率。
- 切出风速：极端大风时停机保护。

可以根据机型功率曲线构造一个理论功率范围：

```text
power_curve_lower_bound(ws)
power_curve_upper_bound(ws)
```

模型预测结果如果严重偏离功率曲线范围，可以进行后处理修正。

注意：功率曲线约束应谨慎使用，因为气象预报风速不一定等于风机轮毂处真实风速，且场站尾流、地形、限电都会影响实际功率。

### 4. 第四层：低风速和高风速规则

可加入简单规则：

- 预报风速极低时，功率预测应接近 0。
- 预报风速处于额定区间时，功率上限接近可用容量。
- 极端大风时应考虑切出风险。

这类规则适合放在后处理阶段，而不是直接替代模型。

### 5. 第五层：爬坡速率约束

功率短时间内变化不应无限快。可以设置：

```text
abs(power_pred(t) - power_pred(t-1)) <= max_ramp_rate
```

不过风电确实可能出现快速爬坡，因此爬坡约束不能过硬。建议用软约束或后处理平滑，避免把真实突变抹平。

### 6. 物理约束加入时机

| 阶段 | 应加入的约束 | 说明 |
| --- | --- | --- |
| 初始建模 | 0 到装机容量裁剪 | 必须加入 |
| 数据口径明确后 | 停机、限电、检修标记 | 决定标签是否正常 |
| 模型稳定后 | 功率曲线、低风速规则 | 用于后处理修正 |
| 部署阶段 | 动态容量、爬坡约束、异常兜底 | 提升业务可靠性 |

## 七、持续校准方法

持续校准不是简单地“预测之后看一下准不准”，而是一个闭环系统：

```text
预测发布 -> 实测回传 -> 误差统计 -> 偏差分析 -> 模型/权重/规则修正 -> 定期重训
```

### 1. 预测发布记录

每次预测都应保存完整记录：

```text
model_version
data_version
init_time
valid_time
lead_time
weather_source
power_pred
post_processed_power_pred
feature_snapshot_id
```

这样等实测功率回来后，才能知道当时模型到底基于什么数据做出的预测。

### 2. 实测回传后误差计算

实测功率回来后，按以下维度计算误差：

- 场站。
- 起报时间。
- 有效时间。
- 预报提前量。
- 气象源。
- 模型版本。
- 风速区间。
- 风向区间。
- 季节/月度。
- 白天/夜间。
- 限电/非限电。
- 停机/非停机。

这些维度可以形成误差分析立方体。

通俗理解：不是只看一个总误差，而是要知道“什么情况下错得最多”。

### 3. 偏差修正

如果发现系统性偏差，可以做后处理修正。

例如：

```text
某气象源在 24-48h 提前量风速偏大 -> 功率预测长期偏高
某风向区间模型预测偏低 -> 该风向区间加校正项
夜间某些月份预测偏高 -> 分月份、分小时修正
```

修正方式包括：

- 分箱均值偏差修正。
- 线性校准。
- Isotonic Regression 校准。
- LightGBM 二级校准模型。
- 多源融合权重调整。

### 4. 滚动重训

建议建立定期重训机制：

| 频率 | 动作 |
| --- | --- |
| 每日 | 计算最新预测误差，生成监控报表 |
| 每周 | 更新校准参数或融合权重 |
| 每月 | 滚动重训主模型 |
| 每季度 | 评估模型结构是否需要升级 |

如果发现数据分布明显变化，例如气象源版本更新、场站设备改造、限电策略变化，应触发临时重训。

### 5. 漂移监控

需要监控：

- 气象变量分布是否变化。
- 功率分布是否变化。
- 风速-功率关系是否变化。
- 缺失率是否变化。
- 各气象源误差是否变化。
- 模型误差是否连续恶化。

如果漂移明显，说明模型可能已经不适合当前数据，需要重新训练或重新校准。

## 八、多模型同步训练与模型集成

### 1. 为什么要多模型

风电预测不建议只依赖一个模型。原因是：

- 不同模型擅长不同模式。
- 不同气象源有不同误差。
- 不同提前量下最优模型可能不同。
- 单模型容易在某些天气过程下失效。

因此企业级方案通常是多模型体系。

### 2. 按气象源训练

可以训练：

```text
EC_LightGBM
HW_LightGBM
PG_LightGBM
EC_XGBoost
HW_XGBoost
PG_XGBoost
```

这样可以比较不同气象源的独立贡献。

### 3. 按预测提前量训练

不同提前量可以使用不同模型：

```text
0-4h：历史功率 + 实测气象 + 最新气象预报
4-24h：气象预报 + 历史功率 + 时间特征
24-72h：多源气象预报为主
72h 以后：趋势参考，更多依赖气象源稳定性
```

短提前量受当前功率影响大，长提前量受气象预报影响大。

### 4. 按模型类型训练

建议同步维护：

- Persistence：短期基线。
- PowerCurve：物理基线。
- LightGBM/XGBoost：主力表格模型。
- RandomForest：稳健对照。
- AutoGluon：快速自动基线。
- NHITS/NBEATSx：深度时间序列基线。
- TFT：多协变量深度模型。
- PatchTST：长窗口序列模型。
- WindFM：风电基础模型探索。

### 5. 集成方法

模型集成可以分阶段：

第一阶段：简单平均。

```text
final_pred = mean(model_1, model_2, model_3)
```

第二阶段：固定加权。

```text
final_pred = 0.5 * LightGBM + 0.3 * XGBoost + 0.2 * NHITS
```

第三阶段：按场景动态加权。

```text
weight = f(lead_time, month, wind_speed, wind_direction, source_uncertainty)
```

第四阶段：Stacking 融合模型。

```text
final_pred = meta_model(base_model_predictions + context_features)
```

其中 `meta_model` 可以用 LightGBM 或线性模型。

### 6. 集成注意事项

- 融合模型训练必须使用时间切分，不能随机打乱。
- 二级模型不能使用未来实测信息。
- 每个基模型要保留独立误差报表。
- 如果某个气象源缺失，要有降级策略。
- 集成后仍要做物理约束和后处理。

## 九、业务考核口径与误差评价方法

### 1. 为什么业务口径重要

模型指标必须和企业或电网考核口径一致。否则模型离线指标好，不代表业务上有价值。

需要企业确认：

- 考核功率点是哪一个。
- 是否考核实际并网功率，还是理论可发功率。
- 限电期间是否参与考核。
- 停机检修期间是否剔除。
- 是否按装机容量归一化。
- 是按 15 分钟点考核，还是按小时平均考核。
- 是否分日前、日内、超短期不同考核。

### 2. 通用技术指标

建议保留：

- MAE。
- RMSE。
- nMAE。
- nRMSE。
- R2。
- Bias。
- 分位数误差。
- 超限次数。
- 大误差时段占比。

### 3. 业务分组指标

建议按以下维度拆分指标：

- 提前量：15min、30min、1h、4h、24h、48h、72h。
- 风速区间：低风速、中风速、高风速、额定附近。
- 风向区间：按 8 或 16 个风向扇区。
- 功率区间：低出力、中出力、高出力。
- 季节/月度。
- 白天/夜间。
- 限电/非限电。
- 停机/非停机。
- 气象源。

总误差只能告诉我们平均表现，分组误差才能告诉我们模型哪里需要改。

### 4. 回测方法

必须使用时间序列回测，而不是随机切分。

建议：

```text
训练：过去若干月
验证：随后一段时间
测试：最后一段时间
```

或者滚动回测：

```text
用 1-3 月训练，预测 4 月
用 1-4 月训练，预测 5 月
用 1-5 月训练，预测 6 月
```

这样更接近真实业务运行。

## 十、推荐模型路线

### 第一阶段：强表格基线

主力模型：

- LightGBM。
- XGBoost。
- RandomForest 作为对照。

输入：

- 多源气象预报。
- 历史功率。
- 历史实测风速风向。
- 时间特征。
- 场站基础特征。
- 状态和质量标记。

建议先做直接预测：

```text
target = power(t+h)
```

然后做残差预测：

```text
target = power(t+h) - baseline(t+h)
```

其中 baseline 可以是 Persistence、功率曲线模型或气象源单模型。

### 第二阶段：多源融合模型

建立单源模型：

```text
EC -> power_pred_EC
HW -> power_pred_HW
PG -> power_pred_PG
```

再建立二级融合模型：

```text
final_pred = f(power_pred_EC, power_pred_HW, power_pred_PG, source_uncertainty, lead_time)
```

这一阶段重点是稳定性，而不是追求单次最优。

### 第三阶段：深度时间序列模型

建议尝试：

- NHITS。
- NBEATSx。
- TFT。
- PatchTST。

其中：

- NHITS/NBEATSx 适合稳健基线。
- TFT 适合历史功率 + 未来气象预报 + 时间特征的多协变量场景。
- PatchTST 适合长窗口序列建模。

深度模型不建议在数据很少时优先使用。等至少有 3 个月以上连续、质量较好的多源气象预报与功率对齐数据后，再系统评估。

### 第四阶段：WindFM 探索

WindFM 可作为技术储备：

- 单独环境。
- 单独数据适配。
- 与 LightGBM/XGBoost/TFT 对比。
- 不作为第一阶段主线。

原因是风电基础模型可能有价值，但依赖和输入格式通常更复杂，且需要验证是否适合企业自己的场站和气象源。

## 十一、训练流程规划

### 1. 数据层

建立统一训练样本表：

```text
station_id
init_time
valid_time
lead_time
power_label
weather_source_features
history_power_features
history_scada_features
station_features
status_flags
quality_flags
```

所有模型都尽量从同一张样本表或同一套特征生成逻辑读取，避免不同模型之间口径不一致。

### 2. 特征层

特征生成应版本化：

```text
feature_version = v1, v2, v3 ...
```

每次模型训练记录使用哪个特征版本。

### 3. 模型层

每次训练记录：

```text
model_name
model_version
train_time_range
valid_time_range
test_time_range
feature_version
data_version
parameters
metrics
```

### 4. 评估层

输出：

- 总体指标。
- 分 horizon 指标。
- 分风速区间指标。
- 分风向区间指标。
- 分气象源指标。
- 分月份指标。
- 典型天气过程误差分析。
- 预测曲线图。
- 大误差样本清单。

### 5. 发布层

预测发布时保存：

- 原始预测。
- 后处理预测。
- 模型版本。
- 输入气象源。
- 预测时间。
- 对应起报时间。

后续实测回来后自动匹配，形成持续校准闭环。

## 十二、关键技术风险

### 1. 数据泄漏

最重要风险是把未来实测数据作为特征。例如：

- 用 `valid_time` 时刻的实测风速预测 `valid_time` 功率。
- 用未来功率滚动均值作为特征。
- 随机切分时间序列数据。

这些都会导致离线指标虚高，但上线后失效。

### 2. 限电和停机污染

如果限电、停机、检修不处理，模型会把非气象原因导致的低功率当成正常规律学习。

建议：

- 有标记时分开建模或剔除。
- 没标记时用异常检测识别疑似限电。
- 评价时单独统计限电样本。

### 3. 气象源版本变化

如果企业气象源模型升级，历史误差规律可能变化。需要记录气象源版本，并监控误差漂移。

### 4. 多源缺失

某些时刻某个气象源可能缺失。系统必须支持：

- 缺失标记。
- 降级到可用气象源。
- 使用最近可靠模型。
- 输出置信度降低提示。

### 5. 指标与业务不一致

如果离线评价口径和业务考核口径不一致，模型优化方向会偏。必须先确认考核口径。

## 十三、阶段性交付建议

### 阶段一：数据口径确认与强基线

目标：

- 确认功率标签。
- 建立多源气象样本表。
- 跑通 LightGBM/XGBoost 多源气象基线。
- 输出按 horizon 的误差报表。

交付：

- 训练样本表。
- 特征说明文档。
- LightGBM/XGBoost 模型结果。
- 多源气象贡献分析。

### 阶段二：残差预测与融合模型

目标：

- 建立 Persistence/PowerCurve 基线。
- 训练残差模型。
- 单源模型分别训练。
- 建立二级融合模型。

交付：

- 残差预测结果。
- 单源模型对比。
- 多源融合模型。
- 分风速、风向、提前量误差分析。

### 阶段三：深度模型实验

目标：

- 训练 NHITS、NBEATSx、TFT、PatchTST。
- 引入未来气象协变量。
- 与树模型和融合模型对比。

交付：

- 深度模型实验报告。
- 多模型对比表。
- 典型天气过程预测曲线。

### 阶段四：持续校准与准上线流程

目标：

- 建立预测发布记录。
- 建立实测回传匹配。
- 建立误差监控。
- 建立定期重训流程。

交付：

- 每日误差监控报表。
- 每周校准报告。
- 每月重训模型版本。
- 模型版本管理和回滚机制。

## 十四、建议优先级

综合当前项目阶段，建议优先级如下：

1. 建立未来气象预报样本表，严格区分 `init_time`、`valid_time`、`lead_time`。
2. 用 LightGBM/XGBoost 跑多源气象预报基线。
3. 做残差预测，重点学习未来功率相对基线的变化。
4. 单源模型分别训练，分析 EC、华为、盘古各自效果。
5. 建立多源融合模型。
6. 加入物理约束和后处理。
7. 建立预测后实测回流的持续校准机制。
8. 数据稳定后再系统做 TFT、PatchTST、WindFM。

## 十五、对外汇报表述

数据全面后，我们建议将模型建设重点放在气象预报驱动的未来几日功率预测上，而不是仅停留在短期功率延续预测。模型体系上不建议从零设计全新模型，而是以 LightGBM、XGBoost、TFT、PatchTST、NHITS、NBEATSx 等成熟模型为基础，结合企业自有功率数据、多源气象预报数据、场站运行状态和业务考核规则进行定制化训练。多源气象数据既可以直接作为模型特征输入，也可以分别训练单源模型后再做融合，并通过多源差异特征刻画预报不确定性。物理约束应分阶段加入，基础阶段保证功率预测不低于 0、不超过装机容量；业务口径明确后进一步加入限电、停机、检修、动态可用容量和风机功率曲线约束。模型上线后需要建立持续校准闭环，即每次预测发布后保存模型版本和输入数据，待实测功率回传后按提前量、风速区间、风向区间、气象源和运行状态统计误差，再修正融合权重、后处理规则，并定期滚动重训。最终目标是形成一套适合企业自身场站、气象源和考核口径的专用预测模型体系。
