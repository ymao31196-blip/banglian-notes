# 风电功率预测实验与未来训练计划汇报

## 一、汇报摘要

本阶段围绕苗庄风电站功率预测，已经完成了从数据理解、样本构造、模型训练到结果评估的初步闭环。当前实验分为两条线：

第一条线是**气象预报驱动的次日功率预测**。该实验使用天津区域气象预报 NetCDF 文件，在苗庄风电站经纬度附近抽取最近气象格点，用未来气象预报变量预测次日 96 个 15 分钟功率点。这条线最接近未来正式业务目标，即利用未来气象预报预测未来几日功率。但目前只有 8 个起报文件，样本量偏少，因此该结果主要用于验证流程，不能作为正式模型精度结论。

第二条线是**短期功率预测实验**。该实验使用 2025 年 1 月至 7 月的实发功率数据，以及 19 台风机实测风速、风向数据，预测未来 15 分钟、30 分钟、1 小时功率。这条线的作用是验证功率口径、时间对齐、特征构造、模型训练和评价流程是否可靠。实验结果显示，短期预测中当前功率延续，也就是 Persistence，是非常强的基线；NeuralForecast 中的 NHITS、NBEATSx 小幅超过该基线，但提升幅度有限。

总体判断是：当前工作已经证明数据处理和建模流程可以跑通；未来工作的重心应转向**气象预报驱动的未来 1 至 3 天功率预测**，并在企业服务器上直接使用全量气象数据和功率数据进行训练，避免全量数据下载到本地。

通俗理解：短期预测像是在判断“现在风电场出力多少，接下来一小会儿大概率也差不多”；未来几日预测则必须依靠“未来天气会怎么变”。因此，正式模型的核心不应停留在历史功率延续，而应转向未来气象预报、多源气象融合和持续校准。

## 二、背景数据情况

### 1. 场站与预测对象

当前实验以苗庄风电站为主要对象。

已知场站坐标为：

```text
经度：117.8406861
纬度：39.40238056
```

当前按台账使用装机容量：

```text
76 MW
```

本阶段预测对象为场站级功率，而不是单台风机功率。当前功率文件中没有单风机功率，使用的是考核点或出线侧功率数据。根据目前理解，同一时间点 311、312、313、314 四路功率相加后代表全场功率；原始值为负值表示发电外送，因此标签定义为：

```text
power_mw = clip(-(311 + 312 + 313 + 314), 0, 76)
```

其中 `clip(0, 76)` 是物理约束，保证功率不小于 0、不超过装机容量。

### 4. 已使用的场站实测风速、风向数据

场站实测数据来自：

```text

data/weather/jk_data/苗庄风电站/25年风向/N01...N19.csv
```

这些数据是 19 台风机的实测风速、风向，原始粒度约为 1 分钟。实验中将其聚合到 15 分钟，与功率数据对齐。

需要强调：这些实测风速、风向只能作为当前及历史状态特征，不能使用未来目标时刻的实测风速、风向，否则会形成数据泄漏。

通俗理解：如果要预测 10:15 的功率，模型最多只能使用 10:00 及以前已经知道的数据，不能使用 10:15 才发生的真实风速。

## 三、已完成的预测实验

### 1. 实验一：气象预报驱动的次日功率预测

#### 实验目的

该实验用于验证未来正式预测流程：

```text
气象预报 NetCDF 文件
-> 苗庄最近格点抽取
-> 气象变量与功率标签按 valid_time 对齐
-> 构造训练样本
-> 训练模型
-> 输出预测结果和评价指标
```

#### 训练任务

每个气象起报文件预测次日 24 小时功率，即：

```text
次日 00:00 至 23:45，共 96 个 15 分钟功率点
```

训练集：

```text
2025-01-01 12:00 至 2025-01-06 12:00 起报文件
```

测试集：

```text
2025-01-07 12:00
2025-01-08 12:00 起报文件
```

样本情况：

| 项目 | 数量 |
| --- | ---: |
| 气象预报理论样本 | 768 |
| 对齐后可用样本 | 763 |
| 缺失功率标签样本 | 5 |
| 训练样本 | 573 |
| 测试样本 | 190 |

#### 模型

使用模型：

```text
RandomForestRegressor
```

对照基线：

```text
DummyMean
```

RandomForest 适合小规模表格数据，可以学习气象变量和功率之间的非线性关系。DummyMean 不看气象，只输出训练集平均功率，用于判断模型是否至少学到有效信息。

#### 结果

| 模型 | MAE(MW) | RMSE(MW) | nRMSE | R2 |
| --- | ---: | ---: | ---: | ---: |
| RandomForest | 25.187 | 28.596 | 37.63% | -1.222 |
| DummyMean | 25.355 | 29.194 | 38.41% | -1.316 |

#### 解释

RandomForest 略优于 DummyMean，说明气象预报变量中确实包含了一些对功率有用的信息。但整体误差仍然较大，R2 为负，不能作为正式精度结论。

主要原因是样本太少。目前只有 8 个起报文件，覆盖不了不同风况、季节变化、限电停机、异常状态等情况。因此这轮实验的价值主要是流程验证，不是精度验证。

### 2. 实验二：短期功率预测树模型基线

#### 实验目的

该实验用于验证功率数据和场站实测风速、风向数据能否支撑稳定训练，并建立短期预测基线。

#### 预测任务

预测目标为：

```text
未来 15 分钟功率
未来 30 分钟功率
未来 60 分钟功率
```

训练集：

```text
2025-01-01 至 2025-05-31
```

验证集：

```text
2025-06-01 至 2025-06-15
```

测试集：

```text
2025-06-16 至 2025-07-01
```

#### 输入特征

主要特征包括：

- 当前功率。
- 历史功率滞后项。
- 功率滚动均值、标准差、最大值、最小值。
- 19 台风机风速聚合特征。
- 风向 sin/cos 特征。
- 风速平方、风速三次方。
- 小时、分钟、星期、年内日周期特征。
- 功率质量标记、运行状态异常计数。
- 风速、风向缺失比例。

#### 模型

已训练模型：

```text
Persistence
DummyMean
LightGBM
XGBoost
```

其中 Persistence 表示：

```text
未来功率 = 当前功率
```

这是短期预测中非常强的基线。

#### 测试集结果

| Horizon | Persistence RMSE | LightGBM RMSE | XGBoost RMSE | DummyMean RMSE |
| --- | ---: | ---: | ---: | ---: |
| 15min | 3.494 | 3.571 | 3.511 | 17.168 |
| 30min | 5.275 | 5.619 | 5.498 | 17.166 |
| 60min | 7.183 | 7.684 | 7.624 | 17.159 |

#### 解释

LightGBM 和 XGBoost 明显优于 DummyMean，说明特征和训练流程有效。但它们没有超过 Persistence，说明短期预测中当前功率惯性非常强。

这不是树模型一定不适合，而是当前实验缺少未来气象预报协变量。短期模型只知道过去和当前风况，但不知道未来风况如何变化，因此很难稳定超过“当前功率延续”。

### 3. 实验三：AutoGluon-TimeSeries

#### 实验目的

AutoGluon-TimeSeries 用于快速验证自动时间序列建模框架能否形成强基线。

#### 实验口径

本轮采用 target-only 方案：

```text
输入：过去 24 小时功率窗口
输出：未来 15min、30min、60min 功率
```

本轮没有加入未来气象预报，也没有使用未来实测风速、风向。

#### 测试集结果

| Horizon | AutoGluon RMSE | Persistence RMSE |
| --- | ---: | ---: |
| 15min | 3.580 | 3.494 |
| 30min | 5.585 | 5.275 |
| 60min | 7.864 | 7.183 |

#### 解释

AutoGluon 本轮没有超过 Persistence。原因主要是输入只包含历史功率窗口，没有未来气象预报变量。它的价值在于快速搭建自动建模框架，后续可在协变量口径稳定后继续使用。

### 4. 实验四：NeuralForecast

#### 实验目的

NeuralForecast 用于验证深度时间序列模型在当前数据上是否能跑通，并与 Persistence 对比。

#### 模型

本轮训练了：

```text
NHITS
NBEATSx
```

输入仍为：

```text
过去 24 小时功率窗口
```

输出为：

```text
未来 15min、30min、60min 功率
```

#### 测试集结果

| Horizon | Persistence RMSE | NHITS RMSE | NBEATSx RMSE | 当前最优 |
| --- | ---: | ---: | ---: | --- |
| 15min | 3.494 | 3.484 | 3.485 | NHITS |
| 30min | 5.275 | 5.245 | 5.227 | NBEATSx |
| 60min | 7.183 | 7.080 | 7.081 | NHITS |

#### 解释

NHITS 和 NBEATSx 在三个预测尺度上都小幅超过 Persistence，说明深度时间序列模型可以从历史功率曲线中提取到一些额外信息。

但提升幅度较小，不能直接作为最终模型结论。更合理的判断是：深度模型路线已经初步跑通，后续需要加入未来气象预报、多源气象特征和更长时间跨度数据后，再评估其真实价值。

## 四、全部实验结果对比

### 1. 短期预测 RMSE 总表

单位：MW，越低越好。

| Horizon | Persistence | LightGBM | XGBoost | AutoGluon | NHITS | NBEATSx | 当前最优 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | --- |
| 15min | 3.494 | 3.571 | 3.511 | 3.580 | 3.484 | 3.485 | NHITS |
| 30min | 5.275 | 5.619 | 5.498 | 5.585 | 5.245 | 5.227 | NBEATSx |
| 60min | 7.183 | 7.684 | 7.624 | 7.864 | 7.080 | 7.081 | NHITS |

### 2. 主要结论

1. 短期预测中，Persistence 是非常强的基线。
2. LightGBM、XGBoost 和 AutoGluon 本轮没有超过 Persistence。
3. NHITS 和 NBEATSx 小幅超过 Persistence，但提升有限。
4. 气象预报驱动的 RandomForest 次日预测已跑通流程，但样本太少，指标不能代表正式效果。
5. 未来正式模型必须以未来气象预报为核心输入。

## 五、当前实验对未来工作的价值

虽然未来重点是预测未来几日，但当前短期预测实验仍有参考价值。

### 1. 验证了功率标签口径

当前已经使用：

```text
power_mw = clip(-(311 + 312 + 313 + 314), 0, 76)
```

完成训练和评估。这个标签口径后续仍会作为主要标签，除非企业确认新的考核口径。

### 2. 验证了时间对齐流程

已经跑通：

```text
功率 15 分钟时间戳
实测风速风向聚合到 15 分钟
气象预报 valid_time 与功率标签对齐
训练/验证/测试按时间切分
```

这些流程可以迁移到未来几日预测。

### 3. 验证了模型评估框架

已经输出：

- MAE
- RMSE
- nMAE
- nRMSE
- R2
- 预测曲线
- 测试集预测表
- 模型结果 Markdown 报告

后续只需要把输入从短期历史特征切换为未来气象预报和多源气象融合特征。

### 4. 暴露了下一阶段重点

短期实验说明：如果没有未来气象预报，复杂模型很难显著超过当前功率延续。因此，未来几日预测不能只靠历史功率，必须把气象预报作为核心输入。

## 六、未来模型选择与训练计划

### 1. 总体方向

未来主线应定义为：

```text
基于未来气象预报的场站级未来 1 至 3 天功率预测
```

模型输入应包括：

- 多源未来气象预报。
- 历史功率。
- 历史实测风速、风向。
- 场站基础信息。
- 时间周期特征。
- 限电、停机、检修、质量标记。
- 多源气象不确定性特征。

模型输出应包括：

```text
每个 init_time 下，多个 valid_time 的功率预测值
```

必须严格区分：

| 字段 | 含义 |
| --- | --- |
| `init_time` | 气象预报起报时间 |
| `valid_time` | 未来有效时间 |
| `lead_time` | 预报提前量 |

### 2. 第一阶段：建立未来气象预报样本表

优先建立统一训练样本表：

```text
station_id
init_time
valid_time
lead_time
power_label
weather_source_features
history_power_features
history_scada_features
station_features
status_flags
quality_flags
```

这个阶段的核心不是调模型，而是把数据口径做稳：

- 每个气象源按 `init_time + valid_time + lead_time` 整理。
- 功率标签按 `valid_time` 对齐。
- 历史功率只能取 `init_time` 之前的数据。
- 历史实测风速、风向只能取 `init_time` 之前的数据。
- 多源气象预报对齐到同一 `valid_time`。
- 缺失气象源保留缺失标记，不简单删除全部样本。

### 3. 第二阶段：强表格模型基线

优先训练：

```text
LightGBM
XGBoost
RandomForest
```

原因是这些模型：

- 对表格特征适配好。
- 训练速度快。
- 可解释性相对较强。
- 适合快速发现数据问题。
- 适合作为长期基线和生产兜底模型。

建议先做直接预测：

```text
target = power(valid_time)
```

再做残差预测：

```text
target = power(valid_time) - baseline(valid_time)
```

其中 baseline 可以是 Persistence、功率曲线模型或单源气象模型。

### 4. 第三阶段：多源气象融合

多源气象包括 EC、华为、盘古等不同来源。使用方式建议分层推进。

#### 直接拼接特征

例如：

```text
EC_ws100
HW_ws100
PG_ws100
EC_wd100_sin / EC_wd100_cos
HW_wd100_sin / HW_wd100_cos
PG_wd100_sin / PG_wd100_cos
```

#### 构造多源统计特征

例如：

```text
ws100_mean
ws100_std
ws100_min
ws100_max
ws100_range
```

这些特征可以表达多个气象源之间的一致性和不确定性。

#### 构造多源差异特征

例如：

```text
EC_ws100 - HW_ws100
PG_ws100 - EC_ws100
EC 与 HW 的风向差异
PG 与 EC 的风向差异
```

差异越大，通常表示未来天气不确定性越高。

#### 单源模型分别训练后融合

建议分别训练：

```text
EC -> EC 功率预测模型
HW -> HW 功率预测模型
PG -> PG 功率预测模型
```

再训练融合模型：

```text
final_pred = f(EC_pred, HW_pred, PG_pred, lead_time, season, source_uncertainty)
```

融合模型可以是加权平均，也可以是 LightGBM/XGBoost 二级模型。

### 5. 第四阶段：深度时间序列模型

在样本量足够后，系统尝试：

```text
NHITS
NBEATSx
TFT
PatchTST
```

其中：

- NHITS/NBEATSx 适合稳健深度学习基线。
- TFT 适合“历史功率 + 未来气象预报 + 时间特征”的多协变量场景。
- PatchTST 适合长窗口序列建模。

深度模型建议在至少有 3 个月以上连续、质量较好的气象预报与功率对齐数据后系统评估。

### 6. 第五阶段：WindFM 探索

WindFM 可以作为风电专用基础模型探索项，但不建议作为第一阶段主线。

原因是：

- 依赖和输入格式可能更复杂。
- 需要单独环境。
- 需要验证是否适合企业自己的场站和气象源。

它可以作为技术储备或对照实验。

## 七、物理约束与业务规则

未来正式模型需要分阶段加入物理约束。

### 1. 初始阶段必须加入

```text
0 <= power_pred <= installed_capacity
```

即苗庄当前：

```text
0 <= power_pred <= 76MW
```

### 2. 数据口径明确后加入

如果企业提供停机、检修、限电、可用风机数量，应加入动态容量约束：

```text
available_capacity = installed_capacity * available_turbine_ratio
0 <= power_pred <= available_capacity
```

同时需要区分：

- 自然可发功率。
- 实际并网功率。

如果限电期间仍使用实际并网功率训练，模型可能会误认为“大风时也应该低功率”。

### 3. 模型稳定后加入

可以加入：

- 风机功率曲线约束。
- 低风速接近 0 出力规则。
- 高风速切出风险规则。
- 功率爬坡速率软约束。
- 预测后平滑和异常修正。

这些规则适合放在后处理阶段，避免一开始规则过多，影响判断模型自身能力。

## 八、持续校准计划

持续校准不是一次性验证，而是长期闭环。

完整流程为：

```text
预测发布
-> 实测功率回传
-> 误差统计
-> 偏差分析
-> 融合权重/后处理规则/模型参数修正
-> 定期重训
```

### 1. 每次预测需要保存

```text
model_version
data_version
feature_version
init_time
valid_time
lead_time
weather_source
power_pred
post_processed_power_pred
```

这样实测回来后才能准确追溯预测来源。

### 2. 误差分析维度

建议按以下维度统计误差：

- 预报提前量。
- 气象源。
- 模型版本。
- 风速区间。
- 风向区间。
- 功率区间。
- 月份/季节。
- 白天/夜间。
- 限电/非限电。
- 停机/非停机。

通俗理解：不能只看一个总误差，而要知道“什么情况下错得最多”。

### 3. 更新频率建议

| 频率 | 动作 |
| --- | --- |
| 每日 | 计算最新预测误差，生成监控报表 |
| 每周 | 更新校准参数或多源融合权重 |
| 每月 | 滚动重训主模型 |
| 每季度 | 评估是否升级模型结构 |

如果气象源版本更新、设备改造、限电策略变化，应触发临时重训。

## 九、远程训练计划

### 1. 为什么需要远程训练

企业侧气象预报数据体量很大。例如天津区域 15 分钟粒度一个月气象预报数据约 30GB，7 天也可能有 7GB 左右。如果全部下载到本地，不利于长期实验和模型迭代。

因此建议采用：

```text
本地小样本开发流程
企业服务器全量训练
只回传指标、报告、模型摘要和必要小文件
```

### 2. 本地阶段

本地主要做：

- 小样本数据查看。
- NetCDF 变量识别。
- 功率标签口径确认。
- 特征生成脚本开发。
- 小批量模型训练。
- 指标和报告模板开发。
- 代码版本管理。

当前本地已经完成了这些核心验证工作。

### 3. 远程服务器阶段

企业服务器上建议做：

- 直接读取 `/data/weather` 下全量气象数据。
- 直接读取企业侧功率、场站、状态数据。
- 在服务器本地完成 NetCDF 抽点、样本构造和模型训练。
- 避免将大体量原始数据下载到本地。
- 只下载训练日志、指标表、Markdown 报告、少量预测样例和必要模型文件。

远程查看和训练方式建议：

```text
SSH key 登录服务器
远程执行 ls/find/ncdump/python 查看数据
远程运行训练脚本
使用 tmux/screen/nohup 保持长任务
训练结果输出到 outputs/ 或统一实验目录
```

### 4. 远程训练环境建议

建议在企业服务器上建立独立环境：

```text
wind-power-baseline
```

或使用容器：

```text
Docker / Conda environment
```

基础依赖：

- Python
- numpy
- pandas
- xarray
- netCDF4
- scikit-learn
- lightgbm
- xgboost
- joblib
- matplotlib

深度学习实验可单独环境：

- NeuralForecast
- PyTorch
- AutoGluon-TimeSeries
- WindFM 相关依赖

建议不要把所有模型依赖堆进同一个环境，避免版本冲突。

### 5. 远程训练执行顺序

建议按以下顺序推进：

1. 在服务器上确认数据目录和文件结构。
2. 远程抽取苗庄坐标附近气象格点。
3. 构造统一样本表。
4. 跑 LightGBM/XGBoost 多源气象基线。
5. 跑残差预测模型。
6. 分别训练 EC、华为、盘古单源模型。
7. 训练多源融合模型。
8. 增加物理约束和后处理。
9. 在数据量足够后训练 TFT、PatchTST、NHITS、NBEATSx。
10. 最后探索 WindFM。

### 6. 远程训练输出

每次远程训练建议输出：

```text
run_summary.json
metrics_by_horizon.csv
metrics_by_weather_source.csv
metrics_by_lead_time.csv
predictions_sample.csv
feature_importance.csv
error_cases.csv
actual_vs_pred.png
训练结果汇报.md
model.joblib 或 model.pt
```

本地只同步：

- Markdown 报告。
- CSV 指标表。
- 少量预测样例。
- 必要模型文件。

原始大体量 NetCDF 不建议同步到 Git，也不建议频繁下载到本地。

### 7. 数据合规与安全

远程训练应遵循：

- 不把企业全量原始数据上传到公开仓库。
- Git 仓库只保存代码、说明文档、小型指标结果。
- 大文件模型、原始数据、预测明细按企业要求存放。
- 服务器访问使用 SSH key，不在文档中记录密码。
- 只在企业许可范围内使用和下载数据。
